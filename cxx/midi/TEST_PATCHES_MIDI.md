# Patch Test MIDI Fixtures

Every patch in `patches/` has a companion MIDI file in this directory.  The
`patch_test` binary uses these together for two purposes:

1. **Smoke test** (`--smoke` flag) — offline render, no audio hardware required,
   suitable for `ctest`.  Verifies the patch produces non-silent audio when the
   MIDI sequence is played.

2. **Audible playback** (default, no `--smoke`) — opens the HAL, renders in real
   time, lets you listen and visually inspect the signal chain.

MIDI files are generated by `tools/gen_test_midi.py`.  Edit that script to
change musical content, then re-run it.  Never edit the `.mid` files directly.

```
python3 tools/gen_test_midi.py          # regenerate all 36 files
python3 tools/gen_test_midi.py --out-dir /tmp/test_mid   # alternate output
```

## How to add a new patch + MIDI pair

1. Create `patches/<name>.json` — define the voice chain.
2. Add a `mk_<name>()` function to `tools/gen_test_midi.py` that returns the
   SMF event bytes.  Document BPM, pattern, and musical intent in a docstring.
3. Add `("<name>", mk_<name>)` to the `PATCHES` list in the script.
4. Run `python3 tools/gen_test_midi.py` to generate `midi/<name>.mid`.
5. Add a smoke test entry to `CMakeLists.txt`:
   ```cmake
   add_patch_smoke(<name>)
   ```
6. Update the correlation table in this file.
7. If the patch has unique signal-processing characteristics worth asserting on
   (e.g. percussive decay ratio, spectral centroid shift, stereo spread), add a
   dedicated `test_<name>_patch.cpp` GTest file for those assertions.

## Suggestions for new patch tests

These are gaps identified during the Phase 27 architecture audit.  Each is a
candidate for both a new patch JSON and a new MIDI fixture.

| Patch concept | Musical content suggestion | Key assertion(s) |
|---|---|---|
| `supersaw_lead.json` (unison mode) | Fast chromatic run, C4–C5 | Spectral centroid > single-VCO equivalent; L/R correlation < 0.7 from spread |
| `paraphonic_strings.json` | 3-note C minor chord | All 3 pitches audible simultaneously in spectrum |
| `fm_bass.json` (VCO fm_in feedback) | Ascending bass line | Spectral centroid increases with pitch |
| `dual_filter_ms20.json` (HP+LP in series) | Mid-register melody | Band-pass frequency response between HP cutoff and LP cutoff |
| `ring_mod_bell_chord.json` | C4 + G4 simultaneous | Sideband frequencies present; fundamentals suppressed |
| `envelope_follower_gate.json` | Rhythmic audio input side-chain | NOISE_GATE opens in sync with input envelope |
| `lfo_filter_sweep.json` | Long sustained note | Spectral centroid oscillates at LFO rate |
| `portamento_lead.json` | Chromatic glide sequence | Frequency transitions smoothly between notes (no step discontinuity) |
| `velocity_sensitive_pad.json` | Same note at vel 30 vs vel 100 | RMS at vel 100 > 2× RMS at vel 30 |

## Smoke test pass criteria

`patch_test --smoke` exits with:
- **0** — patch loaded without error AND peak output amplitude > `1e-4` within
  the rendered duration.
- **1** — patch load failed, MIDI load failed, or output was silent.

The smoke test does not assert on timbre, frequency content, or stereo width —
those belong in the per-patch GTest files.  The smoke test answers one question:
**"Does this patch produce any audio when driven by its companion MIDI file?"**
